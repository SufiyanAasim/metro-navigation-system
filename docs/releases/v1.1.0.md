# Metro Navigation System — v1.1.0

**Codename: Crimson**

## 📖 Overview

Crimson is the first fully-featured, properly licensed, CI-backed release of the Metro Navigation System — consolidating the engine and screens from the Magenta and Neon milestones into a stable, legally-clear, publicly distributable package.

This milestone shifts focus from architecture and layout to project legitimacy: licensing, continuous integration, and packaged distribution, so the project can be shared and built on by others with confidence.

## 🎯 Objectives

- Add an MIT license and proper attribution.
- Stand up GitHub Actions CI to validate every push and pull request.
- Add a PowerShell script to stage and package release binaries.
- Rewrite the README with full setup, feature, and licensing documentation.

## ✨ Added

### 🧭 Shortest Path Calculation
- Dijkstra's algorithm computes the optimal path between any two of the 10 modeled metro stations.

### 🗺️ Karachi Metro Visual Map
- A responsive map interface with active checkboxes/pins representing station selections.

### 🧾 Trip Receipt & History Logging
- Auto-logs trip receipts (date, distance, route) to a local `Metro App Travel History/` directory.

### 🧰 Interactive Options Panel
- Selection controls for map viewing, shortest route, station search, and developer credits.

### 🔊 Tap Sound FX
- Audio feedback on button clicks.

### 🎨 App Icon
- Finalized application icon.

### 📜 MIT License
- The project is now properly licensed and attributed.

### ⚙️ CI/CD
- GitHub Actions workflows validate the build on every push.

### 📦 PowerShell Packaging Script
- `package-release.ps1` stages and zips release assemblies.

## 🔧 Changed

- Rewrote `README.md` with full setup, features, and licensing information.

## 🏛️ Architecture Decisions

**Build automation as a first-class concern.** CI/CD is introduced here rather than bolted on later — every push and pull request is now validated by an MSBuild-based GitHub Actions workflow, catching compile errors before they reach `main`.

**Reproducible packaging.** `package-release.ps1` formalizes what "shipping a release" means: build in Release mode, stage the executable and its runtime dependencies, bundle documentation, and compress into a versioned zip. This same script is used by every release from Crimson onward.

## ⚠️ Known Limitations

- No custom message boxes yet — standard Windows dialogs only (introduced in Teal).
- No automated test suite.
- No multi-language support.
- Windows-only (WinForms / .NET Framework).

## 📌 Development Notes

Crimson's objective was legitimacy, not new functionality: an MIT license, CI validation, and a repeatable packaging process turn the project from a personal prototype into something that can be responsibly open-sourced and built by others. Every release from this point forward goes through the same CI-validated, scripted packaging path.
