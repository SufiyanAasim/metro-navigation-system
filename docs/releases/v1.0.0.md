# Metro Navigation System — v1.0.0

> *Codename: Magenta*

---

## 📖 Overview

Magenta is the first internal development milestone of the Metro Navigation System, laying the technical foundation for a modular, extensible WinForms desktop route planner.

This milestone focuses on establishing the project's form-per-screen architecture, implementing the initial shortest-path engine, and validating the six-screen wizard flow — before introducing licensing, CI/CD, or visual polish. Rather than prioritizing a finished user experience, this release concentrates on building a codebase capable of supporting the licensing, automation, and UI refinement work that follows in later milestones.

The application was designed around a clear separation between screen forms (WinForms `Form` + `.Designer.cs` pairs), the route-calculation engine, and local trip persistence. This separation keeps each screen's code focused on presentation while the engine and storage logic stay reusable and independently testable.

Not intended for public distribution — Magenta transforms the project from an initial concept into a functional software foundation.

---

## 🎯 Objectives

This milestone was developed with the following goals:

* Establish a clean, form-per-screen project structure.
* Implement the initial graph-based shortest path engine.
* Build the six-screen wizard flow end-to-end.
* Validate trip receipt logging to local storage.
* Prepare the project for licensing, CI/CD, and visual refinement in future milestones.

### 🏗️ Project Foundation

Created the initial project structure that separates the WinForms UI, the route engine, and local persistence into dedicated files.

Core source files introduced include:

* `Program.cs` — application entry point
* `Dijkstra.cs` — shortest-path engine (later renamed `MetroNetwork.cs`)

## 🏛️ Architecture Decisions

Several architectural principles were established during this milestone.

**Form-per-screen design.** Each wizard screen is a dedicated WinForms `Form` with a paired `.Designer.cs` file, keeping layout code separate from event-handling logic. This isolates presentation concerns and lets the route engine and persistence layer evolve independently of the UI.

**Local-first storage.** The Metro Navigation System stores all trip data locally as flat text files, with no database, no cloud dependency, and no network calls. This keeps the app fully offline, fast, and simple to deploy — see [Database.md](../Database.md) for how this evolved.

**Extensibility.** The project structure was intentionally kept simple so that licensing, CI/CD, DPI-aware layout, and custom UI components could be layered on in later milestones without requiring a rewrite of the engine or screens.

---

## ⚠️ Known Limitations

As an internal development milestone, several things remain unaddressed:

* No MIT license or CI/CD yet.
* No custom message boxes — standard Windows dialogs only.
* Layout is not yet DPI-aware, and some controls are unpolished.
* Naming inconsistencies not yet cleaned up (`Dijkstra.cs`, `Reciept generation.cs`).
* No packaged release documentation (this is the first version to receive one).

These are addressed progressively in subsequent releases.

## 📌 Development Notes

The primary objective of Magenta is architectural validation rather than end-user polish.
